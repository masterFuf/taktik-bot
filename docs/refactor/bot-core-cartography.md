## Regles deja deduites de la cartographie

- `shared/device/**` est la source de verite pour la connexion device, l'ATX et les primitives Android partagees.
- Ne pas recreer `device/` comme compat import boundary : les consommateurs internes ont ete migres.
- Une facade plateforme peut vivre sous `social_media/<platform>/.../device/` si elle ajoute un sens metier plateforme ; un `DeviceManager` a cet endroit doit etre un shim ou une specialisation explicite, pas une troisieme implementation generique.
- Une aide DB historique reste acceptable temporairement dans une plateforme seulement comme facade de compat documentee ; toute nouvelle logique SQL doit vivre sous `database/**`.

## Lots proposes

1. Lot 1 - boundary `device/` : termine. Le shim legacy a ete retire apres migration des derniers consommateurs internes vers `shared/device/**`.
2. Lot 2 - ownership DB Instagram + messaging : sous-lots faits pour `processed / filtered / skip`, `processed_hashtag_posts` et `unfollow sync`. Le SQL `following_sync` / `followers_sync` vit maintenant dans `repositories/instagram/social_graph/`, la famille `unfollow` appelle deja `InstagramFollowGraphService` directement, la famille `hashtag` appelle deja `InstagramHashtagPostService` directement, le runtime Instagram (business + UI extractors) appelle maintenant `InstagramWorkflowStateService` directement pour `processed / filtered / skip`, `workflows/post_scraping/post_persistence.py` ne porte plus de SQL inline sur `instagram_profiles`, et le scraping Instagram partage maintenant une dependency `local_db` explicite entre `scraping_workflow`, `persistence`, `list_scraping` et `deep_qualify` au lieu de multiplier les `get_local_database()` caches dans chaque mixin. `sent_dms` est maintenant porte par `repositories/messaging/SentDMRepository`, avec `database/messaging.py` comme facade service et `bridges/common/database.py` comme compat. Le shim `DatabaseHelpers` a ete supprime apres audit des imports internes.
3. Lot 3 - managers runtime : `instagram/core/manager.py` et `tiktok/core/manager.py` pointent maintenant vers `shared/platform/social_media_base.py` et `shared/device/manager.py` comme `threads/core/manager.py`, au lieu de repasser par les shims Instagram. Le framework selectors/versioning de `compat` vit maintenant sous `compat/selectors/**`, les bridges internes pointent vers ces owners directs, et les anciens modules top-level ont ete retires. `clone/packages/package_map.py` devient la source de verite pour les package names officiels et prefixes de clone, puis `clone/**` est splitte par owner (`detection`, `packages`, `device`, `selectors`).
